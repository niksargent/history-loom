# Public Language Guide

## Purpose

Set the language standard for the public-facing product.

The History Loom is not written for academic historians only.
Every public surface should pass the `16-year-old test`:
- can a normal teenager understand it quickly
- does the wording feel concrete rather than abstract
- does the page reveal something before it starts explaining itself

## Core rule

Internal precision and public clarity are not the same thing.

The product should keep both:
- an internal analytical vocabulary
- a public-facing vocabulary

Public UI should prefer:
- concrete
- visual
- short
- vivid
- human-readable

## What to avoid in public copy

Avoid:
- technical builder language
- academic hedging in hero positions
- abstract compound labels that require decoding
- system-explaining text the user did not ask for

Examples to avoid:
- `derived readings from the pressure field`
- `editorial prompts from the data layer`
- `historical causation`
- `mobilised settlement`
- `structural family`

These may be acceptable in docs or internal review.
They are poor default UI language.

## What to prefer

Prefer wording that points to a felt pattern:
- `order rebuilt under pressure`
- `fast change, fraying trust`
- `forces that rise together`
- `moments that break the pattern`
- `eras that feel alike`
- `power hardens`
- `trust falls away`
- `new technology outruns old rules`

## Public writing principles

### 1. Revelation first

Lead with what the user can notice, not with how the system works.

Better:
- `Patterns worth noticing`

Worse:
- `Derived patterns from the pressure field`

### 2. Concrete before abstract

Better:
- `order rebuilt after crisis`

Worse:
- `restabilising settlement dynamics`

### 3. Short before complete

The page does not need to say everything.
It needs to say enough to make the user curious.

### 4. Show, then explain

If a visual can carry the meaning, keep the copy lighter.

## Translation model

Where the product still needs analytical precision, use:
- internal label
- public label
- optional short plain-language explainer

Example:
- internal: `mobilised settlement`
- public: `order rebuilt under pressure`

## Whole-tool scope

This applies across:
- Insights Lab
- Loom labels
- Force Explorer
- Compare
- Detail panel
- entry cards
- empty states
- helper text

## Acceptance test

Before shipping a public label or heading, ask:
1. Would a 16-year-old understand this in one pass?
2. Does the phrase sound like a human explanation rather than a model note?
3. Can some of the meaning be carried by colour, layout, or motion instead?

If the answer is no, rewrite it.
